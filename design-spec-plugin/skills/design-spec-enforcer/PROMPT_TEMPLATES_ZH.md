# SaaS 设计提示模板

这些模板是给日常直接提需求用的中文实战版。把方括号里的内容替换掉，基本就能直接发给 Codex。

## 1. 通用页面生成模板

```md
请使用我的 SaaS Design Spec Plugin，为 [产品/功能] 设计一个 [页面类型]。

页面目标：
- [主要业务目标]

目标用户：
- [目标用户]

必须包含的模块：
- [模块 1]
- [模块 2]
- [模块 3]

主要操作：
- [主要操作]

次要操作：
- [次要操作]

重点数据或状态：
- [指标、字段或状态]

要求：
- 必须遵循本地 SaaS design spec 和最接近的页面模式。
- 页面层级清晰，风格偏企业级平台。
- 信息密度可以偏高，但要易于扫读。
- 需要考虑加载中、空状态、错误状态。
- 最后总结本次设计应用了哪些规则。
```

## 2. 工作台首页模板

```md
请基于我的 SaaS design spec，设计一个 [产品/团队] 的工作台首页。

页面需要包含：
- 顶部关键指标概览
- 公告/通知模块
- 快捷操作入口
- 最近动态或常用内容模块
- 一个和 [业务重点] 强相关的运营模块

要求：
- 按工作台首页模式来设计。
- 优先参考 workplace 类型页面的区块骨架：欢迎/上下文、概览、快捷操作、内容或动态模块。
- 页面要像用户每天登录后的第一站。
- 兼顾概览信息和可执行操作。
- 整体气质要克制、高效、像成熟平台产品。
```

## 3. 监控页模板

```md
请基于我的 SaaS design spec，设计一个 [系统/流程] 的监控页面。

页面需要包含：
- 当前状态概览
- 指标卡片
- 告警或消息列表
- 快捷操作区
- 一个协同/沟通相关模块

要求：
- 按监控页模式来设计。
- 要突出当前状态和异常信息，但不能做成“满屏告警墙”。
- 页面必须支持高频查看，扫读效率要高。
```

## 4. 数据分析页模板

```md
请基于我的 SaaS design spec，设计一个 [业务领域] 的数据分析页。

页面需要包含：
- KPI 概览卡片
- 趋势图区域
- 对比分析区域
- 明细拆解区域
- 结论/解读模块

要求：
- 按数据分析页模式来设计。
- 优先参考 data-analysis 类型页面的区块顺序：KPI、主趋势、对比分析、拆解、结论。
- 图表要放在干净的卡片中，标题和辅助信息要清楚。
- 颜色使用要克制，优先服务信息表达。
- 页面重点是帮助做判断，而不是追求视觉噱头。
```

## 5. 搜索表格页模板

```md
请基于我的 SaaS design spec，设计一个 [实体对象] 的搜索表格页。

页面需要包含：
- 筛选/搜索表单
- 表格工具栏
- 主表格
- 行级操作
- 批量操作

要求：
- 按搜索表格页模式来设计。
- 优先参考 search-table 类型页面的区块顺序：标题、可选 KPI、筛选区、工具栏、表格、分页。
- 筛选必须放在表格上方。
- 不要把筛选区做成写死的 1 行或 2 行，要根据字段数量和优先级自动决定布局。
- 如果筛选条件很多，优先保证可读性和分组逻辑，再决定是否做 2 到 3 行或“更多筛选”折叠。
- 表格工具栏要和筛选区分开，批量操作、刷新、列设置、密度、导出不要混成一组。
- 页面可以紧凑，但必须容易扫读。
- 状态、标签、操作优先级要一眼能看懂。
```

## 6. 卡片列表页模板

```md
请基于我的 SaaS design spec，设计一个 [实体对象] 的卡片列表页。

页面需要包含：
- 顶部筛选和排序
- 重复卡片列表
- 状态标识
- 轻量级卡片操作

要求：
- 按卡片列表页模式来设计。
- 如果这个页面同时适合表格视图和卡片视图，要保持筛选逻辑和排序逻辑一致，只切换结果展示层。
- 卡片结构要统一，不要每张卡长得都不一样。
- 不要为了“丰富”而牺牲浏览效率。
```

## 7. 分组表单页模板

```md
请基于我的 SaaS design spec，设计一个 [业务流程] 的分组表单页。

页面需要包含：
- 页面标题和上下文说明
- 多个分组表单区块
- 主提交按钮
- 次要操作按钮
- 校验和提示文案

要求：
- 按分组表单页模式来设计。
- 对齐要整齐，分组节奏要清晰。
- 要降低长表单填写疲劳感。
```

## 8. 分步表单页模板

```md
请基于我的 SaaS design spec，设计一个 [业务流程] 的分步表单页。

页面需要包含：
- 步骤条
- 当前步骤核心内容
- 上一步 / 下一步操作
- 最终确认摘要

要求：
- 按分步表单页模式来设计。
- 进度感要清楚。
- 单步内容不要过载。
```

## 9. 资料页 / 设置页模板

```md
请基于我的 SaaS design spec，设计一个 [用户/对象] 的 [资料页/设置页]。

页面需要包含：
- 顶部摘要区
- 分组信息区块
- 安全/权限模块（如果相关）
- 认证/状态模块（如果相关）
- 清晰稳定的编辑入口

要求：
- 按最接近的资料页或设置页模式来设计。
- 如果是设置页，优先参考 user/setting 类型页面：摘要、信息、安全、认证/状态分组。
- 分区必须清楚，方便浏览和编辑。
- 页面既要适合查看，也要适合修改。
```

## 10. 成功页 / 错误页 / 异常页模板

```md
请基于我的 SaaS design spec，设计一个 [成功页/错误页/异常页]，场景是 [具体场景]。

页面需要包含：
- 清晰的状态图形或状态图标
- 简洁说明文案
- 一个主要下一步操作
- 一到两个次要操作

要求：
- 按结果页或异常页模式来设计。
- 页面保持简洁、稳定、不过度装饰。
- 恢复路径必须明确。
```

## 11. 设计审查模板

```md
请按我的 SaaS design spec 审查这个页面。

重点检查：
- 页面层级是否清楚
- 导航和壳层结构是否合理
- 间距节奏是否统一
- 卡片和表面样式是否一致
- 按钮、输入框、状态样式是否统一
- 语义色是否用得正确
- 是否符合最接近的页面模式

输出要求：
- 先说符合的部分
- 再说不符合的部分
- 最后给出优先级最高的修改建议
```

## 12. 组件生成模板

```md
请基于我的 SaaS design spec，设计一个 [组件名称]。

使用场景：
- 所属页面/流程：[页面或流程]
- 主要作用：[作用]
- 需要覆盖的状态：[状态]
- 信息密度：[紧凑/平衡]

要求：
- 必须遵循本地 SaaS design spec。
- 复用已有的圆角、层级和语义状态逻辑。
- 组件要像系统里原生长出来的，不要像单独拼出来的。
```
